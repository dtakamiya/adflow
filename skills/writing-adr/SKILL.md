# ADR作成スキル

## アクティブなワークフロー
!`for d in docs/*/; do f=${d}.adflow-context.md; if [ -f $f ]; then echo "--- $(basename $d) ---"; head -20 $f; echo; fi; done 2>/dev/null || echo "アクティブなワークフローなし"`

## 既存ADR一覧
!`ls docs/*/*-adr.md 2>/dev/null || echo "ADRファイルなし"`

## 鉄則（絶対ルール）

1. **最低2つの選択肢を検討する** — 1つの案だけで決定しない。比較なき決定はADRではない。
2. **銀行固有4セクションを空欄にしない** — トランザクション影響・監査要件・データ分類・規制影響は必ず具体的に記入する。「該当なし」の場合もその理由を明記する。
3. **既存ADRとの矛盾を放置しない** — 新しいADRが既存の決定と矛盾する場合、既存ADRのステータスを「Superseded」に更新する。
4. **推測で書かない** — 不明な点はユーザーに確認する。特に規制影響とデータ分類は推測禁止。

## Red Flags — よくある合理化

| 思考 | 現実 |
|------|------|
| 「選択肢は1つしかない」 | 必ず代替案がある。「現状維持」も選択肢の1つ |
| 「銀行固有セクションは後で埋めよう」 | 後で埋まることはない。今書く |
| 「この決定は小さいからADRは不要」 | 小さな決定の積み重ねがアーキテクチャを形作る |
| 「既存ADRは読まなくていい」 | 矛盾した決定はシステム障害の原因になる |
| 「規制影響は開発者の範囲外」 | 規制を無視した設計は後で全面書き直しになる |

## 引数の処理

- `$ARGUMENTS` が提供された場合: その内容をADRのタイトルとして使用し、kebab-case変換して機能名ディレクトリ名とする。Step 1 のコンテキスト収集に進む
- `$ARGUMENTS` が空の場合: ユーザーにADRのタイトルと対象を確認してからStep 1に進む

**機能名ディレクトリ名の決定:**
- タイトルから英語のkebab-caseに変換する（例: 「振込機能」→ `transfer-service`、「口座管理」→ `account-management`）
- `docs/` ディレクトリをスキャンし、既存の機能ディレクトリから最大の連番（NNNN）を取得し、それに+1したものを新しいディレクトリのプレフィックスとする（例: `0001-transfer-service`）
- このディレクトリ名を `{dir-name}` として以降のステップで使用する

## ワークフロー

### Step 1: コンテキスト収集と要件分析

**1-1. 現状把握（自己実行）:**
ユーザーの指示に基づき、まずは以下の調査をエージェント自身で実行する:
- 関連する既存のコードベース・依存関係の特定と読み込み
- 過去のADR (`docs/*/*-adr.md`) の参照
- プロジェクトルールの確認

**1-2. ユーザーへの確認:**
調査後、不足している情報があればユーザーに以下を確認する:
1. **決定の対象**: 何についてのアーキテクチャ決定か
2. **背景**: なぜこの決定が必要になったか
3. **制約**: 技術的・ビジネス的制約はあるか
4. **検討した選択肢**: どのような選択肢を検討したか（最低2つ）

### Step 2: 機能ディレクトリ作成とADR採番

1. `mkdir -p docs/{dir-name}` でディレクトリを作成する
2. `docs/` ディレクトリ配下をスキャンし、既存のADRファイル名（または親ディレクトリ名）から最大番号を取得する（パターン: `NNNN-*.md` または `NNNN-*`）
3. 次のADR番号を4桁ゼロ埋めで生成する（例: 0001, 0002, ...）

**エラーハンドリング:**
- ADRが1つもない場合は `0001` から開始する

### Step 3: テンプレート適用

1. `templates/adr-template.md` を読み込む
2. ユーザーの入力に基づいてテンプレートを埋める
3. 銀行固有セクションを必ず記入する:
   - **トランザクション影響**: この決定がトランザクション処理に与える影響
   - **監査要件**: 記録すべき監査イベント
   - **データ分類**: 取扱データの分類とPII有無
   - **規制影響**: 関連する金融規制

**エラーハンドリング:**
- `templates/adr-template.md` が見つからない場合、ユーザーに報告し以下のビルトイン構造で代替する:
  - タイトル、ステータス、コンテキスト、決定、結果、銀行固有セクション（トランザクション影響・監査要件・データ分類・規制影響）

### Step 4: ADR生成

1. `docs/{dir-name}/01-adr.md` にファイルを作成する
2. ファイルのタイトルは英語のkebab-caseとする

### Step 5: ADRインデックス更新

1. `docs/{dir-name}/README.md` が存在しない場合は作成する
2. 新しいADRをインデックスに追加する:
   ```markdown
   | {NUMBER} | {TITLE} | {STATUS} | {DATE} |
   ```

### Step 6: AI自己レビュー（自動品質チェック）

生成したADRに対して、AI自身で以下の品質を独自検証する:
- [ ] ステータスが設定されている
- [ ] 最低2つの選択肢が記述されている
- [ ] 各選択肢にメリット・デメリット・リスクがある
- [ ] 銀行固有の4セクションがすべて記入されている
- [ ] プロジェクトの既存アーキテクチャや制約と矛盾していないか
- [ ] 関連ADRへのリンクがある（該当する場合）

問題があれば、ユーザーに提示する前に自律的に修正を行う。

### Step 7: 承認ゲート（HARD-GATE — 人間によるバリデーション）

自己レビューで品質を満たしたADRをユーザーに提示する。

**ゲート条件チェックリスト:**
- [ ] ステータスが設定されている
- [ ] 最低2つの選択肢が比較されている
- [ ] 銀行固有4セクションがすべて記入済み
- [ ] 既存ADRとの矛盾がない
- [ ] AI自己レビューの全項目がパス

**ユーザーアクション:**
- 「承認」→ Step 8 に進む
- 「修正」→ 指摘箇所を修正して再提示
- 「スキップ」→ ⚠️ **警告: ADRの承認をスキップすると、以降の設計・実装の根拠が不明確になります。本当にスキップしますか？**

### Step 8: コンテキストファイル作成

1. `templates/adflow-context-template.md` を読み込む
2. `docs/{dir-name}/.adflow-context.md` に書き込む
3. 以下の内容を記録する:
   - 機能名: `{dir-name}`
   - 現在のステージ: `spec`
   - ADRの成果物: ファイルパス `docs/{dir-name}/01-adr.md`、ステータス「完了」
   - その他のステージ: ステータス「未着手」
4. ユーザーに案内する: **「次のステージ `/spec` を実行するか、`/clear` して後で再開できます」**
   - **重要**: `/workflow` からの実行でない限り、自律的に次のコマンドを実行してはいけません。必ずユーザーからの明示的なコマンド実行（または進行の承認）を待ってください。
