# 銀行コードレビュースキル

## 銀行レビューチェックリスト
!`cat ${CLAUDE_SKILL_DIR}/bank-review-checklist.md 2>/dev/null || echo "チェックリストなし"`

## Git状態
!`git diff --name-only --cached 2>/dev/null; git diff --name-only 2>/dev/null`

## 引数の処理

- `$ARGUMENTS` が提供された場合: その内容をレビュー対象のブランチ名またはファイルパスとして使用する
- `$ARGUMENTS` が空の場合: 上記の「Git状態」に基づき、現在の変更差分をレビュー対象とする。変更がない場合はユーザーに対象を確認する

## エラーハンドリング（前提チェック）

- Gitリポジトリでない場合: 「このディレクトリはGitリポジトリではありません」とエラーを報告して中止する
- 変更がない場合（`git diff` が空）: 「レビュー対象の変更がありません」と通知する

## ワークフロー

### Step 1: 変更差分の取得

1. `git diff` で変更差分を取得する
2. ステージング済みの変更: `git diff --cached`
3. 特定ブランチとの差分: `git diff {base-branch}...HEAD`
4. 変更ファイル一覧を取得: `git diff --name-only`

### Step 2: 一般品質チェック

以下の観点でレビューする:
- コードの可読性と命名
- 関数の長さ（50行未満推奨）
- ファイルの長さ（800行未満推奨）
- ネストの深さ（4階層以下）
- エラーハンドリングの適切さ
- SOLID原則の遵守
- DRY原則の遵守

### Step 3: 銀行固有チェックリスト適用

上記で読み込んだ「銀行レビューチェックリスト」に基づき、以下のカテゴリでチェック:

#### CRITICAL（リリースブロッカー）
- トランザクション安全性
- 金銭計算の安全性（BigDecimal使用）
- 監査ログの実装

#### HIGH（マージブロッカー）
- 排他制御の実装
- 冪等性の確保
- 入力バリデーション
- エラーハンドリング

#### MEDIUM（改善推奨）
- データ保護（PII暗号化）
- パフォーマンス（N+1クエリ等）

### Step 4: セキュリティチェック

`references/financial-security-checklist.md` を参照して:
- SQLインジェクション
- XSS
- 認証・認可の漏れ
- シークレットのハードコード

### Step 5: レビュー結果の出力

以下のフォーマットで出力:

```markdown
# コードレビュー結果

## サマリー
- 変更ファイル数: {N}
- 追加行数: {N} / 削除行数: {N}
- CRITICAL指摘: {N}件
- HIGH指摘: {N}件
- MEDIUM指摘: {N}件

## CRITICAL 指摘事項
### [{ファイル名}:{行番号}] {指摘タイトル}
- **問題**: {問題の説明}
- **リスク**: {このまま残した場合のリスク}
- **修正案**: {具体的な修正方法}

## HIGH 指摘事項
{同様のフォーマット}

## MEDIUM 指摘事項
{同様のフォーマット}

## 良い点
- {良い実装の指摘}
```

### Step 6: 修正の支援

CRITICALおよびHIGHの指摘について:
1. 具体的な修正コードを提案する
2. **必ずユーザーの承認を取得してから** 修正を適用する
3. 修正後にテストを再実行する
